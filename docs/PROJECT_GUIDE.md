# Agroforestry Carbon Field Guide

## Project Guide (Version 0.1)

> This document captures the guiding principles for the project. It is
> the source of truth for editorial, research, and design decisions
> expected to remain stable over time.

------------------------------------------------------------------------

# 1. Mission

The Agroforestry Carbon Field Guide exists to help readers understand
the organizations, business models, and relationships that make up the
agroforestry carbon ecosystem.

The site is intended to teach the ecosystem, not simply catalog
organizations.

# 2. Audience

-   Climate finance practitioners
-   Agroforestry project developers
-   Development finance professionals
-   Corporate sustainability teams
-   Researchers and students

Assume readers are intelligent but may be unfamiliar with the structure
of the market.

# 3. Design Philosophy

The website is a **Field Guide**, not a company directory.

Every page should help readers answer:

1.  What does this organization do?
2.  Where does it fit?
3.  What organizations are commonly confused with it?
4.  How is it different?
5.  Where can I learn more?

The objective is orientation first, detail second.

# 4. Organization Profile Standard

1.  Hero
2.  Primary Role
3.  Closest Adjacent Organizations
4.  Featured Comparison
5.  Research Profile
6.  Common Collaborators
7.  References

# 5. Ecosystem Philosophy

Organizations are organized by ecosystem role rather than popularity.

Featured comparisons are editorial choices designed to teach, not rank.

Closest Adjacent Organizations identify organizations readers are most
likely to confuse.

# 6. Editorial Standards

-   Hero summary: 75--125 words.
-   Ecosystem Fit: 2--3 concise sentences.
-   Adjacent comparison: 1--2 concise sentences.
-   Narrative sections use short paragraphs with blank lines.
-   Representative projects use bullets whenever practical.
-   References prioritize official sources.

# 7. Research Standards

-   Official sources first.
-   Never invent information.
-   Distinguish documented facts from inference.
-   Preserve existing fields when expanding profiles.

# 8. Technical Workflow

1.  Research one organization.
2.  Validate JSON.
3.  Review the diff.
4.  Commit one profile per commit.
5.  Review the live page.
6.  Update this guide whenever standards change.

# 9. Living Document

This guide should evolve with the project. When a new standard is
adopted, document it here so future contributors and coding assistants
inherit the project's philosophy.
